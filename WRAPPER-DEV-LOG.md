## 2025-09-02（Wrapper API 接続問題・FFmpeg 書き込み失敗の整理）
- 決定事項
  - REST の音声入力は“コンテナ付き”のバイト列をそのまま Backend `/asr` に送る（wav/mp3/m4a/webm 等）
  - 拡張子 `.raw` の場合のみ 16kHz/mono/PCM16 とみなし、ラッパー側で WAV に包んでから送信
  - OpenAI Whisper API 互換のレスポンス（json/text/srt/vtt/verbose_json）とエラー形式（OpenAI 風 JSON）を実装
  - `model` は必須だが値は無視。GUI（Backend）側のモデル設定を使用する

- 根拠（観測ログ）
  - クライアント: `[LocalWhisperAPI error] 500: {"message":"timed out during opening handshake"}`
  - サーバ: 500 の直後に `WebSocket /asr [accepted]` → 即切断（タイムアウト後に遅れて accept したレース）
  - `Error writing to FFmpeg: Connection lost` が多発（raw PCM を送ると FFmpeg `-i pipe:0` が解釈できず早期終了）
  - `Empty audio message received`/`ran 0 times`（実データが届かず終端のみ到達）

- 未解決事項（要対応）
  - Wrapper → Backend 接続の耐性強化：`websockets.connect(..., open_timeout=30)` とリトライ（例: 3 回、指数バックオフ）
  - 起動直後のモデルロード待ちに備えたリトライ戦略の設計（API レイテンシの安定化）
  - `/v1/audio/translations` の完全互換（Backend がリクエスト単位で `task=translate` を受けられない）
  - リクエスト単位の `language`/`prompt`/`temperature` の反映（WS メタメッセージの検討を upstream 提案）
  - 逐語タイムスタンプ（word 粒度）対応（`verbose_json` 拡充。token 単位情報の WS 出力追加を upstream 提案）

- 次アクション（Wrapper 側で実施可能）
  - Backend 接続に `open_timeout` と簡易リトライ導入（実装案済）
  - 接続先環境変数の README 追記（`WRAPPER_BACKEND_*` の具体例と注意点、IPv4 明示）
  - 最初の `/v1/audio/transcriptions` 実行前に Backend ヘルスチェック（`GET /` で疎通確認）を検討
  - 入力検証の強化（空ファイル検知、`multipart/form-data` キー名チェックの明確化）

- リスク
  - open_timeout 延長によりハングに見える時間が伸びる可能性（ただしリトライ併用で緩和）
  - `.raw` を 16kHz/mono と仮定して WAV 化するため、異なる仕様の raw を渡された場合にタイミングがずれる可能性
  - word 粒度のタイムスタンプ非対応は、完全互換を期待するクライアントで差異となる

- 運用メモ（環境変数）
  - `WRAPPER_BACKEND_HOST`（例: `0.0.0.0`）
  - `WRAPPER_BACKEND_PORT`（例: `63916`）
  - `WRAPPER_BACKEND_CONNECT_HOST`（`0.0.0.0` 指定時は `127.0.0.1` を明示）
  - `WRAPPER_BACKEND_SSL`（`1`=wss / `0`=ws）
  - Windows 例（PowerShell）: `setx WRAPPER_BACKEND_CONNECT_HOST 127.0.0.1`

— 以上を踏まえ、Wrapper 側は接続リトライと README のトラブルシュート追記を優先。Backend 側のリクエスト単位パラメータ適用は upstream 提案に記録。

## 2025-09-01 (FFmpeg書き込みエラー: GUIレコーダー時のフォーマット不一致)
- 背景・スコープ：GUIのRecorder使用時に FFmpeg "Connection lost" が連発し処理が進まない事象。
- 決定事項：README-FOR-WRAPPER.md に原因/回避策を明記（トラブルシューティングを追加）。
- 根拠・検討メモ：
  - 上流FFmpegは -i pipe:0 でコンテナ音声を想定。Web UIは audio/webm（正常）。GUIレコーダーは生PCM16を送信→自動判別不能でFFmpegが早期終了。
  - 参照: whisperlivekit/ffmpeg_manager.py:62, whisperlivekit/web/live_transcription.js:388, wrapper/app/gui.py:1943
- 未解決事項：GUIレコーダー経路での入力指定切替（-f s16le -ac 1 -ar 16000 -i pipe:0）または webm 送信化。
- 次アクション：ラッパー側でフォーマット整合の実装案を検討し、upstream非改変で解決（起動引数/送信形式の切替）。
- リスク・課題：ブラウザ外でwebmエンコードを行う場合の依存増、あるいはFFmpeg起動モードの分岐の複雑化。
- 参照リンク：README-FOR-WRAPPER.md（本節）

## 2025-09-04 (ウィンドウ縮小時にカラム高さが戻らない)
- 背景・スコープ：ウィンドウを拡大後に縮小すると左右カラムの高さが維持され、UI 全体が隠れる問題。
- 決定事項：ScrollableFrame を改修し、子ウィジェットの実高さから再計算して縮小時にも高さがウィンドウに追随するようにした。
- 根拠・検討メモ：
  - canvas bbox では拡大後の高さが保持されるため、子ウィジェット位置から自然な高さを算出する方式に変更。
- 未解決事項：なし
- 次アクション：ユーザーからのフィードバック監視。
- リスク・課題：プラットフォームごとの Tk 挙動差異。
- 参照リンク：README-FOR-WRAPPER.md（GUI 層の更新）
## 2025-09-05

- 決定事項: GUIセクション（スクロール領域）の縦幅がウィンドウサイズを超えないようにし、GUIセクションの最小高さをウィンドウ最小高さに連動させる。
- 根拠: 小さい画面で縦方向に内容が溢れ、ウィンドウ外へはみ出す事例があったため。スクロールにより全体を閲覧でき、かつ最小レイアウトが破綻しないことを保証する。
- 変更内容:
  - `wrapper/app/gui.py`: `content.grid(..., sticky="nsew")` とし、可視高さに追従。
  - その後のレビューで、固定的な最小サイズ設定が過剰と判断されたため撤廃。
    - ルートウィンドウ `minsize` 固定（1200×800）とスクロール行 `minsize` の連動設定を削除。
    - PanedWindow のペイン `minsize`（左800/右360）も削除。
  - `README-FOR-WRAPPER.md` に仕様追記。
- 未解決事項: 端末やテーマ差によるヘッダー実高さの差分は `winfo_height()/winfo_reqheight()` で吸収するが、ピクセル単位の誤差が出る可能性あり。
- 次アクション: 実機での最小ウィンドウ時の余白確認、必要ならパディング補正。
- リスク: 非標準テーマにおける `winfo_*` 値の差異でスクロール領域最小高さが過小/過大になる可能性。

- 決定事項: Start/Stop API ボタンをヘッダーへ移動し、メイン2カラム設定画面を折りたためるトグルを追加。折りたたみ状態は保存・復元。
- 根拠: 主要操作（Start/Stop）を常にアクセスしやすくし、設定領域を使わない運用時の視認性を高める。
- 変更内容:
  - `wrapper/app/gui.py`: ヘッダーに Start/Stop を配置、左端に折りたたみボタン追加（`_toggle_main_sections`）。`settings_collapsed` を設定ファイルに保存。
  - 右カラムの比率を Recorder:Logs=3:4 に変更し、トランスクリプト欄の高さを体感で約 2/3 に抑制。
  - `README-FOR-WRAPPER.md` に設計変更を反映。
